Async ETL Pipeline & Data Lake
этот пет-проект  полноценная data engineering инфраструктура написанная мною за 3 вечера, построенная на паттерне producer-consumer.
пайплайн асинхронно выкачивает данные из внешнего API, делает бекапы в щзеро данных (Data Lake), кидает задачи в очередь и аккуратно складывает очищенные данные в реляционную бд.

все автоматизированно в docker, для лёгкого запуска.

поясню как работает под капотом:
я распилил логику на два независимых асинхронных процесса:

producer: качает сырой JSON из внешнего API по расписанию (через apscheduler + aiohttp).
data lake: сразу делает бэкап сырого файла в minIO (через aioboto3), чтобы исторические данные не терялись.
очередь: кидает задачу на обработку в Redis.
сonsumer: сидит в фоне, мгновенно вытаскивает данные из редиса, чистит их и грузит в PostgreSQL.

чуть про фишки:

слабая связность (decoupling): продюсер и консьюмер не ждут друг друга. если база данных временно ляжет, скрипт не упадет - родюсер продолжит складывать сырые данные в minIO и копить задачи в очереди Redis.
идемпотентность: запись в Postgres идет через INSERT ... ON CONFLICT DO NOTHING. дубликатов не будет, даже если прогнать один батч 10 раз.
агрегация: сделал аналитическую витрину, которая сама обновляет стату при новых данных (ON CONFLICT DO UPDATE).
отказоустойчивость: обработка сетевых ошибок — если API отвалится, скрипт просто скипнет цикл и не крашнется.

как запустить?

скопируйте env.example и подставьте значения.
склонируйте репозиторий с помощью команды в терминале: docker compose up -d --build.

админка minIO будет доступна на http://localhost:9001, а логи воркера можно посмотреть через docker compose logs -f etl_worker.
